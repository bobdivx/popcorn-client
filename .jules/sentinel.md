## 2025-02-27 - Implicit Trust of Backend-Rendered HTML
**Vulnerability:** The `DescriptionPreview` component was implicitly trusting and rendering raw HTML injected by the backend (e.g., descriptions) via `dangerouslySetInnerHTML` without any prior validation or sanitization, introducing a critical Cross-Site Scripting (XSS) vulnerability.
**Learning:** In this codebase, data described as 'already rendered by the backend' cannot be implicitly trusted for safety on the frontend, especially if it may contain untrusted user input from an upstream source (e.g., tracker descriptions).
**Prevention:** Always use `isomorphic-dompurify` to sanitize backend-provided HTML content on the frontend before injecting it with `dangerouslySetInnerHTML`. The isomorphic variant guarantees safety across both Client-Side Rendering and Server-Side Rendering (SSR) environments, preventing hydration mismatches and SSR vulnerabilities.

## 2025-02-27 - Weak PRNG Usage for Persistent Identifiers
**Vulnerability:** Weak pseudo-random number generation (`Math.random()`) was being used as a fallback for generating cryptographic-like unique identifiers (UUIDs) and persistent device IDs when standard APIs were unavailable or during mock initialization. `Math.random()` is cryptographically insecure and predictable.
**Learning:** In isomorphic environments like this project (Astro + Tauri), PRNG fallbacks can accidentally default to insecure implementations if environmental checks fail or are not exhaustive enough. We must ensure a hard failure rather than a silent insecure fallback for persistent or security-sensitive identifiers.
**Prevention:** Always use `globalThis.crypto.getRandomValues()` for generating random bytes, and strictly throw an error (`throw new Error("No secure random number generator available")`) if a secure generator cannot be found. Never fall back to `Math.random()` loops.
